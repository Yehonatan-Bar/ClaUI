Happy Coder - Simplification Plan

Background: The Happy Coder integration was built with an overly complex approach -- a custom WebSocket client, cryptographic authentication module, and dedicated message handling system (6 new files, 2 new libraries). In practice, the Happy CLI is simply a wrapper around the existing Claude CLI -- it accepts the same commands and produces the same output. This means we can reuse 100% of the existing infrastructure.

Before / After

Before (current state):
  • 6 new custom files
  • 2 new library dependencies
  • Custom WebSocket connection
  • Custom cryptographic authentication
  • Custom message translation layer
  • Limited features (no forking, no compacting, no images)
After (planned):
  • 0 new files
  • 0 new dependencies
  • Reuses existing Claude infrastructure
  • Only change: which CLI to launch
  • Full feature support (everything Claude can do)
  • Authentication via terminal QR code

Implementation Steps

1 Delete unused files -- Remove the 6 overengineered files and 2 library dependencies that are no longer needed.
2 Allow alternative CLI path -- Add a small change to the process manager allowing it to run happy instead of claude. A single configuration point.
3 Route "Happy" provider to regular sessions -- When the user selects "Happy" in the dropdown, create a regular session tab that uses the happy CLI. All existing features (streaming, tools, cost tracking, forking, etc.) work automatically.
4 Add authentication command -- New command "ClaUi: Authenticate Happy Coder" opens a terminal where the user can scan a QR code for initial authentication.
5 Update UI labels -- Rename "Remote" to "Happy" in the provider dropdown and all display text.
6 Update settings -- Remove old unused settings (server URL, reconnect, keepalive). Add one new setting for the Happy CLI path.
7 Build, deploy, test -- Verify everything compiles, install the extension, and test the flow.
Key insight: Since the Happy CLI wraps Claude CLI and produces identical output, the entire existing chat system, streaming, tool handling, cost tracking, session history, and forking all work automatically. No need to rebuild any of it.

User Experience

  1. User selects "Happy" in the provider dropdown
  2. Clicks "Start Session" (or it auto-starts)
  3. If first time: shown a message to run "ClaUi: Authenticate Happy Coder" from the command palette
  4. Terminal opens with QR code for scanning
  5. After scanning: authentication is saved permanently
  6. Session starts and works exactly like a regular Claude session, but connected to Happy Coder
