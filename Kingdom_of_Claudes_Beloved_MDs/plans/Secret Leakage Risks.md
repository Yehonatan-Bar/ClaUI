# Deep Research Report on Secret Leakage Risks in Agentic Coding Tools

## Executive summary

The strongest practical finding is that secrets and sensitive data do **not** leak only through the obvious “prompt text sent to the model” path. In current agentic coding tools, the highest-risk channels are automatic context expansion, shell and terminal output capture, screenshots/browser traces, MCP/tool calls, cloud or remote agent runtimes, Git artifacts, local history/config exports, and telemetry or diagnostics. That pattern appears repeatedly across official docs, security advisories, and user reports for GitHub Copilot, Cursor, Claude Code, Codex, Continue, Cline, Roo Code, Aider, OpenHands, and Devin. citeturn12view3turn12view2turn14view0turn14view2turn14view3turn14view4turn10search1turn33view1turn35view0turn35view2turn5view1turn16view1turn19view0turn21view0turn21view1turn22search2turn32view2

A second finding is that vendor-native controls are inconsistent and should not be treated as a primary security boundary. GitHub explicitly documents that Copilot content exclusions do **not** apply to Copilot CLI, Copilot cloud agent, or Agent mode in IDEs. Cursor documents that requests still go through its backend even when a user supplies their own API key, and its advisories show multiple prompt-injection-to-execution or exfiltration chains. Codex documents that its own hook interception is incomplete for some shell paths and WebSearch, while Claude Code documents that model prompts/outputs, optional telemetry, and opt-in feedback can all leave the machine. citeturn12view3turn14view4turn14view0turn14view1turn14view2turn14view3turn35view3turn10search1turn33view1

For a VS Code wrapper extension, the correct design is therefore **multi-boundary DLP**, not “prompt filtering.” The wrapper should independently inspect and control at least five boundaries: what is about to be sent to a model, what files or selections are about to be exposed, what command is about to execute, what tool/terminal/MCP/browser output is about to be fed back into the agent loop, and what data is about to be persisted to logs/history/exports/Git/remotes. Anything less will miss the most realistic leakage paths already documented in the field. citeturn5view1turn19view0turn18view0turn21view0turn22search2turn33view4turn35view3

The most defensible default policy is also clear. Hard secrets should be blocked from leaving the workstation or entering Git, telemetry, MCP, screenshots, or diagnostics. Sensitive business data and internal operational data should usually be redacted or require explicit, time-limited approval. Ordinary source code should usually be allowed, but only after path-based and content-based scanning, because code repositories frequently contain `.env` files, private keys, connection strings, internal URLs, and customer data mixed into tests, fixtures, docs, and generated output. citeturn12view3turn16view1turn21view0turn21view1turn22search4turn32view3

## Leakage taxonomy and threat model

A practical leakage taxonomy for your extension should be organized by **boundary crossed**, not by file type. The same secret can leak through a prompt, a terminal error, an MCP response, a screenshot, or a Git comment. The incidents above show that the most important patterns are: prompt/paste leakage, automatic file/context expansion, terminal stdout/stderr echo, diagnostic/history export, model-generated command execution, MCP request/response exfiltration, Git artifact publication, screenshot/browser capture, and persistent state reuse in cloud or snapshot-based environments. citeturn5view1turn19view0turn18view0turn21view0turn22search2turn32view2turn35view0turn10search1

| Leakage pattern | Typical data | Typical path | Default policy |
|---|---|---|---|
| Secret in pasted prompt or selected code | API keys, JWTs, passwords, private keys | Prompt content, clipboard, chat history | **Block** |
| Secret-bearing file auto-attached | `.env`, config exports, creds files, key material | File context expansion, indexing, repo scan | **Block** |
| Agent opens unrelated sensitive files | Internal docs, prod configs, database dumps | Autonomous file discovery | **Require approval** or **block** if path is protected |
| Secret printed in terminal output | Tokens, connection strings, internal URLs | Shell output, stack traces, login flows | **Mask**, then **require approval** if large or repeated |
| Agent-generated command modifies security control files | `.git/**`, `.cursor/**`, `.claude/**`, MCP configs, hooks | File write or shell | **Block** by default |
| MCP request or response carries sensitive context | Internal docs, logs, secrets, query results | Third-party tool transport | **Require approval** or **block** if unsanctioned |
| Browser or screenshot captures private state | PII, support data, dashboards, cookies in visible state | Browser automation, screenshot attachments | **Require approval** and **mask where possible** |
| Agent pushes or comments secrets to Git | Tokens, PII, internal architecture, hidden reasoning | Commit, PR, issue, review, branch push | **Block** |
| Telemetry or diagnostics store sensitive context | Prompts, outputs, file names, command lines | OTel, Sentry, PostHog, JSONL, bug reports | **Mask** or **block** depending on content |
| Untrusted content injects instructions | Hidden issue comments, README text, rules/skills files, webpages | Prompt injection into agent loop | **Warn** or **require approval** before any external or write action |
| Cloud/snapshot inheritance overexposes state | Shared org secrets, cookies, env vars, cloned repos | Cloud VM, golden snapshots, remote sandboxes | **Require approval** and scope secrets per task |

**Assets to protect.** Your extension should treat four asset sets as first-class: hard secrets; sensitive business/user data; internal operational topology; and “ordinary” source/doc/config assets that become sensitive when they embed secrets, PII, internal URLs, production logs, or confidential business logic. It should also protect secondary assets that many teams forget: approval state, agent control files, local extension storage, chat history files, exported settings bundles, and browser/session state. citeturn16view1turn21view0turn22search2turn32view2turn33view0

**Trust boundaries.** At minimum, there are nine: user/editor; your wrapper; the wrapped agent UI or CLI; the model provider; remote or cloud runtimes; shell/subprocess execution; MCP servers and connectors; Git remotes and collaboration platforms; and telemetry/logging/diagnostic backends. A single request can cross several of these in one loop. For example: prompt → model → command → terminal output → model → MCP → Git comment. That chain is exactly why single-point “prompt scanning” is insufficient. citeturn12view1turn12view2turn10search1turn33view1turn35view0turn35view5

**Entry points.** The practical entry points are: chat prompt text, pasted clipboard content, selected code, auto-attached files, repository indexing, issue/PR text, README/rules/skills files, terminal output, browser pages, screenshots, MCP resources, exported settings, history files, telemetry, and remote app-server sockets. Several of those were explicitly implicated in current advisories or bug reports. citeturn5view1turn14view0turn14view2turn14view3turn21view0turn28view1turn35view1

**Attacker types.** The realistic attacker set includes: a malicious repository author; a malicious issue/PR commenter; a malicious or compromised MCP server vendor; a malicious webpage or documentation page; a compromised package or setup script; an insider who pastes too much or applies overly broad secret scope; and a benign user who simply does not realize how much context the agent is sending. Most observed cases so far are design bugs, insecure defaults, or accidental over-sharing rather than sophisticated nation-state tradecraft. That is exactly why a wrapper must optimize for preventing ordinary mistakes. citeturn8search18turn12view2turn14view1turn19view0turn21view1turn27view2

**Highest-risk kill chains.** The most dangerous chain is “untrusted content + autonomous execution + external destination.” That is the pattern documented by Copilot’s prompt-injection guidance, Cursor’s advisories, OpenHands’ prompt-injection analysis, and Codex’s internet-access documentation. The next most dangerous chain is “broad inherited state + cloud/runtime autonomy,” illustrated most clearly by Devin snapshots and by cloud-agent architectures generally. citeturn12view2turn14view0turn14view1turn14view2turn27view2turn32view2turn35view0

## Recommended filtering architecture

Your extension should be implemented as a **policy broker** that sits between the user, the editor, and every outbound or execution boundary that the wrapped agent can reach. The core idea is simple: convert every meaningful boundary crossing into a structured event, scan it, decide an action, and record a secret-safe audit entry. This architecture is necessary because current tools have documented blind spots or limitations in their own controls. Copilot agentic modes bypass content exclusions; Cursor’s backend remains in-path; Claude Code and Codex both have optional telemetry/logging paths; Codex hook coverage is explicitly incomplete for some shell and WebSearch paths; Roo custom tools can be auto-approved; and Cline/Continue show that hidden context expansion and verbose diagnostics are real. citeturn12view3turn14view4turn33view1turn35view3turn21view1turn5view1turn18view0

A workable architecture has eight stages:

| Stage | What the wrapper intercepts | What it should do | Default action |
|---|---|---|---|
| Context manifest | User prompt, pasted text, selected code, attached files, auto-discovered files | Build an exact manifest of bytes, paths, line ranges, screenshots, URLs, MCP resources, and destinations before anything is sent | No send until manifest exists |
| Pre-send scanner | Prompt text plus attached context | Regex + entropy + keyword + path heuristics + allowlist/denylist + repo policy labels | Block, mask, warn, or approval |
| File exposure scanner | Any file opened for model context, indexing, or MCP upload | Scan content and also classify path trust level: secret files, internal docs, generated logs, dump files, control files | Hard block for protected paths |
| Command preflight | Proposed shell commands, package manager commands, Git commands, browser opens | Parse semantically; classify env mutation, credential discovery, external POST, control-file writes, network use, destructive ops | Approval or block |
| Post-tool / output scanner | Terminal stdout/stderr, MCP responses, browser text, screenshots, web results | Redact secrets before they re-enter the model loop; cap size; replace with secure summaries when needed | Mask by default; approval on sensitive data |
| Git publication scanner | Diffs, commits, PR titles/bodies, issue comments, review comments, branch names | Rescan all publication artifacts before commit/push/post | Block on hits |
| Persistence guard | Local history, exported configs, JSONL, bug reports, telemetry, OTel, cloud sync | Prevent raw secret/sensitive content from being written locally or exported | Mask or block |
| Audit and exception engine | Every decision | Record only rule IDs, hashes, destinations, actor, timestamp, exception scope, and byte counts redacted | Always on |

A **project-level policy file** should drive most of the decisions. In practice, that file should define protected paths, internal domains, allowed model providers, allowed MCP servers, allowed Git remotes, blocked commands, approval-required command classes, PII sensitivity thresholds, and exception scopes. It should also let security teams declare repository-specific zones such as `prod/`, `snapshots/`, `customer-data/`, `support-dumps/`, and agent control directories. This matters because the same file extension can be harmless in one repo and highly sensitive in another. That is a lesson repeated by Continue’s whole-file context expansion, Roo’s plaintext settings exports, and Aider’s `.env` conventions. citeturn5view1turn21view0turn22search4

A **destination-aware decision engine** is more important than a simple sensitivity score. The same internal hostname in local-only processing may be acceptable, but the same hostname sent to a cloud provider or a public issue comment is not. Similarly, a hard secret should usually be blocked no matter the destination, but PII or operational metadata may be allowed to a local model and blocked to remote MCP servers or telemetry. Cursor, Claude Code, Continue, and Codex all document distinct outbound paths that need different treatment. citeturn14view4turn10search1turn16view0turn35view2

Use a **stable redaction format** rather than silently dropping content. A good format is something like:

```text
<REDACTED type="AWS_SECRET_ACCESS_KEY" source=".env" hash="sha256:8a3f…" length="40" />
```

This lets the agent continue reasoning while making it obvious that material was removed. For user/business data, a schema-like redaction is better:

```text
<REDACTED type="EMAIL_ADDRESS" count="27" source="prod-log-snippet" />
```

The wrapper should optionally provide a secure, local-only side panel that shows the human what was redacted and why, without ever putting the raw value back into the model path.

A **temporary exception system** is necessary, but it must be narrow. Exceptions should be bound to all of: a repo, a destination, a rule ID, a path pattern, a user, and a short expiry. “Allow this once for this MCP server and this exact file” is a good exception. “Disable secret scanning for the next week” is not. Exceptions should also never override certain invariants: private keys, well-formed cloud credentials, Git hook writes, agent-control-file writes, and public publication of secrets should remain unoverrideable hard blocks.

A **per-tool risk profile** should drive default friction:

| Risk tier | Typical tool posture | Default policy |
|---|---|---|
| Low | Local-only read/chat with no shell, no browser, no MCP, no telemetry | Allow code silently after scan; audit minimal metadata |
| Moderate | Local agent with file edit but no shell/network | Allow most source files after scan; approval for broad context or internal docs |
| High | Local shell agent, Git write access, package manager access | Preflight every command; scan all stdout/stderr before reinjection; scan diffs before apply |
| Very high | Browser-enabled, MCP-enabled, network-enabled, cloud agent | Approval for all external domains and remote tools; stricter redaction; no screenshots or auth pages without consent |
| Critical | Background cloud agent with shared org secrets, cookies, or inherited snapshots | Task-scoped secrets only; no silent operation; full audit; strongest policy set |

The **recommended default policy** for a real organization is:

- **Hard block:** hard secrets; private keys; `.env*`; auth/cookie databases; exported plaintext settings bundles; writes to `.git/**` and agent-control files; agent-originated external POST/PUT/PATCH/DELETE; publication of any detected secret to Git/comments/issues; unsanctioned MCP servers.
- **Redact automatically:** secrets in terminal output, logs, bug reports, screenshots metadata, JSONL streams, and telemetry; common PII in logs when the user asked for generic coding help.
- **Warn and require approval:** internal URLs/IPs, prod logs, database extracts, screenshots of authenticated pages, large file auto-attachments, MCP access to internal systems, cloud-agent indexing or background runs.
- **Allow but audit:** ordinary source files, tests, docs, and read-only local workflows after scan.
- **Allow silently:** only low-risk local workflows with no policy hits and no remote boundary crossing.

## Concrete detection rules

The rule set below is tuned to the kinds of data and leak paths repeatedly seen in the sources above: `.env` files and credential stores, plaintext exported settings, verbose terminal and error output, database connection strings, MCP credentials, cloud-snapshot state, and internal operational identifiers. These are **sample implementation rules**, not canonical vendor grammars. Formats evolve, so the wrapper should version rules and test them continuously. citeturn16view1turn21view0turn21view1turn22search4turn19view0turn32view3

| Rule | What it detects | Sample pattern or heuristic | Suggested action | False-positive risk | Implementation notes |
|---|---|---|---|---|---|
| Private key blocks | PEM/OpenSSH/PGP private keys | `-----BEGIN (?:RSA |EC |OPENSSH |PGP |DSA )?PRIVATE KEY-----` | **Block** | Low | Treat as non-overrideable. Also block adjacent metadata and filenames like `id_rsa`, `*.pem`, `*.p12`, `*.jks`. |
| GitHub PATs and tokens | Common GitHub token formats | ``\b(?:ghp|gho|ghu|ghs|ghr)_[A-Za-z0-9]{20,255}\b`` | **Block** | Low | Also scan issue comments, PR bodies, and commit messages before publish. |
| OpenAI / Anthropic / generic API keys | Common provider API keys | Example only: ``\bsk-[A-Za-z0-9_-]{20,}\b`` and ``\bsk-ant-[A-Za-z0-9_-]{20,}\b`` | **Block** | Medium | Keep a provider-specific rule pack. Do not rely on one regex because formats change. |
| AWS access key pairs | AWS key IDs and nearby secrets | Key ID: ``\b(?:AKIA|ASIA|ABIA|ACCA)[A-Z0-9]{16}\b`` plus nearby 40-char secret and AWS keywords | **Block** | Medium | Pairing dramatically reduces FPs. Also protect `~/.aws/credentials` and `~/.aws/config`. |
| GCP service-account material | Service-account JSON and embedded private key | JSON containing `"type":"service_account"` and `"private_key":"-----BEGIN PRIVATE KEY-----` | **Block** | Low | Also protect `application_default_credentials.json`. |
| Azure and cloud connection strings | Storage and service connection strings | Example: `DefaultEndpointsProtocol=https;AccountName=...;AccountKey=...;EndpointSuffix=...` | **Block** | Medium | Prefer parser-based detection over regex when possible. |
| Database credentials and DSNs | Connection strings with credentials | ``(?i)\b(?:postgres(?:ql)?|mysql|mongodb(?:\+srv)?|redis|amqp|mssql)://[^/\s:@]+:[^@\s]+@[^/\s]+`` | **Block** or **redact** | Medium | Reduced FP if host is non-example and secret chars are present. |
| JWTs and bearer tokens | Auth tokens in logs and headers | JWT example: ``\beyJ[A-Za-z0-9_-]{5,}\.[A-Za-z0-9._-]{10,}\.[A-Za-z0-9._-]{10,}\b`` | **Mask** in logs/output; **block** if outbound | Medium | Many JWT-like examples exist in docs/tests; lower severity in test files or `example` fixtures. |
| Slack/webhook/signing secrets | Webhooks and signing URLs | Slack webhook example: ``https://hooks\.slack\.com/services/[A-Za-z0-9/_-]+`` | **Block** | Low | Also detect `SIGNING_SECRET`, `WEBHOOK_SECRET`, and similar keywords nearby. |
| High-entropy generic secrets | Unknown token formats | Entropy threshold on long base64/hex/urlsafe strings, but only when near keywords like `key`, `secret`, `token`, `password`, `bearer`, `credential` | **Warn** or **block** depending on destination | High | Essential for custom org secrets. Pair entropy with context keywords and exclude known-safe hashes/checksums. |
| Sensitive file paths | Known secret-bearing files and dirs | Path heuristics: `.env`, `.env.*`, `.npmrc`, `.pypirc`, `.netrc`, `.docker/config.json`, `.kube/config`, `.aws/`, `.ssh/`, `terraform.tfstate`, `secrets*.json`, `credentials*` | **Block** | Low | Path rules catch dangerous content before file-read costs are incurred. |
| Agent and Git control files | Control-plane reconfiguration | `.git/**`, `.gitconfig`, `.git/hooks/**`, `.cursor/**`, `.claude/**`, `.codex/**`, `.roo/**`, `.continue/**`, `.cline/**`, `.vscode/**` | **Block** unless explicit admin-approved workflow | Low | This rule exists because config-file and Git-hook attacks are already documented. |
| Internal URLs and service names | Internal operational topology | RFC1918 IP ranges; host suffixes like `.corp`, `.internal`, `.svc`, `.cluster.local`; keywords like `prod`, `staging`, `bastion`, `rds`, `vpn`, `jumpbox` | **Warn** or **require approval** | Medium | Allow `localhost`, `127.0.0.1`, and obvious test fixtures by default. |
| PII in logs and dumps | Emails, phone numbers, IDs, ticket text | Density-based rule: multiple emails/phones/IDs in a short window, or filenames like `users.csv`, `support_dump`, `prod-log` | **Mask** or **require approval** | Medium to High | Single email in source code may be harmless. Density and file-context outperform raw regex alone. |
| Large-output risk | Oversized terminal/MCP/browser output likely to contain mixed sensitive data | Byte or token threshold plus source type, e.g. `>8KB` terminal output or large MCP/database result | **Require approval** | Medium | Before approval, run a summary extractor locally and present a safe synopsis rather than the raw blob. |
| Selection mismatch | Actual send scope exceeds visible user intent | Selected range is small, but attached context includes whole file, many files, or screenshots | **Require approval** | Low | This is the control that would have caught Continue-style whole-file expansion. |
| External network exfil primitives | Agent is trying to send data off-box | Command classifier for shell/network actions with outbound body or upload semantics | **Block** by default | Medium | Use parser-based classification; avoid brittle blacklist-only approaches. |
| Screenshot and attachment heuristics | Visual leaks that text scanners miss | Authenticated domain, browser screenshot count, OCR-like text density, presence of emails/order numbers/account IDs | **Require approval** | Medium | Full OCR is expensive and imperfect; start with metadata and destination-domain rules. |

A few implementation choices matter more than the exact regexes.

First, **scan before and after execution**. Pre-execution catches dangerous commands and file reads. Post-execution catches the secret that only appeared in stdout, a browser page, or a tool response. Cursor, Codex, Claude Code, and Cline all show why both sides are needed. citeturn14view0turn33view4turn35view3turn19view0

Second, **classify by destination**. A JWT in a local buffer and a JWT in a GitHub issue comment are different risk events. The same goes for internal URLs, which may be acceptable for a local-only model and unacceptable for a cloud agent or third-party MCP server. citeturn12view2turn14view4turn35view0

Third, **prefer path heuristics before full content scanning** for obviously sensitive locations. Blocking `.env`, `.ssh`, `.aws`, exported settings bundles, and agent-control files is cheaper, simpler, and usually more reliable than trying to score their contents after the fact. citeturn21view0turn21view1turn22search4turn14view1turn14view3

## UX recommendations and implementation roadmap

The user experience needs to make hidden context visible. The documented failures are often not “the user intentionally leaked a secret,” but “the tool sent more than the user realized.” Continue attached whole files unexpectedly, Cline surfaced long hidden environment details in error output, and Copilot agentic modes ignore exclusions that admins may assume are active. Your wrapper should therefore show an explicit **outbound manifest** before risky sends: destination, model/provider, MCP server if any, file list, line ranges, screenshot count, browser domains, and whether the send is local-only or cloud-routed. citeturn5view1turn18view0turn12view3

A good approval dialog should answer four questions, in plain language: **what is leaving**, **where it is going**, **why it was flagged**, and **how to proceed safely**. Do not show a generic “Sensitive content detected.” Show: “This request would send `.env.production` and a Slack webhook URL to a remote model provider.” Then offer safe actions: redact and continue, remove file from context, open local-only mode, or approve once for this file and this destination. This reduces prompt fatigue while keeping decisions intelligible.

For terminal and tool output, the safest UX is often **automatic redaction with a visible note**, not a modal. If a test command printed a JWT or a connection string, replace it before it re-enters the model loop and show a concise notice such as: “1 auth token redacted from terminal output.” Save modals for cases where the output is large, repeated, or contains business data such as customer records or prod logs.

For screenshots and browser use, the wrapper should be stricter. Show the domain, whether the page is authenticated, how many screenshots will be attached, and whether text extraction is enabled. Browser captures should default to approval-required, because screenshot leaks are difficult to detect perfectly and frequently include mixed sensitive data.

An **audit view** is essential in organizational deployments. It should not store raw secrets. It should store event hashes, rule IDs, path categories, destination classes, redaction counts, approver identity, and exception expiry. Security teams care about patterns such as “Team X keeps trying to send support dumps to remote models” or “An unsanctioned MCP server was attempted in Repo Y.” They do **not** need the raw token value preserved in the log.

A realistic implementation roadmap is:

| Phase | Scope | Why it should come first |
|---|---|---|
| Initial foundation | Pre-send prompt scanning, file-path protection, terminal command preflight, terminal output redaction, Git diff/commit scanning, secret-safe audit log | These controls directly address the highest-confidence and most frequently documented leak paths. |
| Hardening | MCP proxying and request/response scanning, screenshot/browser gating, settings export/import protection, telemetry/history guards, policy file support | This closes the major “non-prompt” paths that most wrappers initially miss. |
| Enterprise maturity | Central policy distribution, org-level allowlists, exception workflow, SIEM export, per-tool risk profiles, shadow-mode tuning, incident replay tests | Needed for real deployment in mixed-tool organizations. |
| Advanced assurance | Local summarization for oversized outputs, invisible-character scanning, rules/skills/repo instruction review, network egress integration, signed custom tools/MCP trust model | These features reduce subtle prompt-injection and supply-chain risk. |

## Open questions and limitations

Public evidence almost certainly **understates** the real volume of leakage and near-miss events. Some vendors publish advisories, some bury security-relevant behavior in product docs, and many enterprise incidents never become public. So this report should be read as a **lower bound** on the problem, not an upper bound. citeturn14view0turn14view1turn14view2turn14view3turn12view3turn10search1turn35view0

A VS Code wrapper alone is not a perfect containment boundary. If an unrelated extension or CLI can directly open sockets or send data to its own backend, the wrapper can only provide strong protection for workflows it actually brokers or intercepts. For high-assurance environments, you should assume you will also need process allowlisting, extension allowlisting, and outbound network policy enforcement. This is an inference from current architectures, especially tools that explicitly route through vendor backends or run cloud agents. citeturn14view4turn12view2turn10search1turn35view1

Screenshot and visual leak detection remains an unsettled area. The sources make clear that screenshots and browser automation are real risk channels, but robust local redaction without introducing heavy OCR, privacy, or performance costs remains difficult. That area deserves more dedicated testing and probably its own risk mode rather than being treated as “just another attachment.” citeturn19view0turn27view2turn32view2

Some tool ecosystems in your requested list have less public, high-confidence security reporting than others. In this report, the strongest evidence was available for Copilot, Cursor, Claude Code, Codex, Continue, Cline, Roo Code, Aider, OpenHands, and Devin. Windsurf and some adjacent agentic tools almost certainly share many of the same architectural risks, but public documentation and advisories were thinner in the material gathered here, so recommendations for them should be treated as pattern-based extrapolation rather than incident-based proof.
