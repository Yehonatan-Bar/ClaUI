★ מה ולמה

כיום, לאחר כל משימת פיתוח, נוצר מסמך תיעוד (SR-PTD) באופן אוטומטי. עם הזמן מצטברים עשרות מסמכים כאלה. קיימת כבר מערכת (Python toolkit) שמנתחת את המסמכים ויוצרת מהם Skills עבור Claude -- אבל היום זה דורש הפעלה ידנית.

הפיצ'ר החדש: המערכת תספור מסמכים חדשים, תציג מונה ויזואלי, ובהגעה לסף מוגדר -- תריץ את הצנרת בסביבת עבודה מבודדת (staging/workspace), תבצע בדיקות תקינות, תבדוק כפילויות/שדרוג מול Skills קיימים, ותתקין בצורה בטוחה רק את מה שחדש או משודרג.

⚙ רכיבים מרכזיים

  • מונה מסמכים (SkillGenStore) -- לא רק מונה: גם ledger מתמשך של fingerprints למסמכים (path/mtime/size או hash תוכן), סטטוס "ממתין לעיבוד", והיסטוריית הרצות. מאופס/מעודכן רק אחרי install מוצלח.
  • מנהל הצנרת (SkillGenService) -- הלב של המערכת. סורק מסמכים, בודק סף, מבצע preflight, אוכף נעילה בין חלונות/תהליכים, מפעיל את הצנרת, ומנהל state machine של ריצה (queued/running/failed/succeeded).
  • מריץ Python (PythonPipelineRunner) -- מריץ את מערכת ה-Python הקיימת כתת-תהליך או דרך Python API. תומך ב-resume (כשמשתמשים ב-run_pipeline.py / .pipeline_progress.json), מנטר התקדמות, timeout וביטול.
  • מנוע כפילויות (DeduplicationEngine) -- בודק בשלוש רמות האם Skill חדש כבר קיים, דורש שדרוג, או שהוא באמת חדש -- עם עדיפות לבדיקות דטרמיניסטיות (traceability.json/fingerprints) לפני AI.
  • מתקין (SkillInstaller) -- מתקין מ-staging ליעד בצורה אטומית ככל האפשר, מגבה קיימים לפני שדרוג, ותומך rollback במקרה שכשל באמצע התקנה.

🔎 מנגנון בדיקת כפילויות

לפני התקנת כל Skill שנוצר ב-skills_out (staging), המערכת בודקת האם כבר קיים Skill דומה/זהה. כדי להימנע מטעויות ועדכונים שגויים, התהליך עובד בשלוש שכבות (דטרמיניסטי קודם, AI אחרון):

שכבה 1: מקור/Traceability

השוואת traceability.json, fingerprint של חבילת ה-Skill, וחפיפת מסמכי מקור. אם זהה -- דילוג מלא (idempotent). אם יש חפיפה חלקית -- מועמד לשדרוג.

שכבה 2: מטא-דאטה + תוכן

השוואת שם, frontmatter, תיאורים, כותרות ומונחים מרכזיים (ללא AI). ספי הדמיון הם התחלתיים בלבד ודורשים כוונון עם נתונים אמיתיים (לא hardcode סופי כברירת מחדל).

שכבה 3: AI

למקרים לא ברורים, שימוש ב-Claude Sonnet לקבלת החלטה: האם מדובר באותו נושא עם יכולות חדשות, או בנושא שונה? החלטת AI לא מוחקת/דורסה Skills בלי גיבוי ובלי traceability תקין.

⚙ הגדרות חדשות

  • הפעלה/כיבוי -- הפעלה או כיבוי של כל הפיצ'ר
  • סף מסמכים -- כמה מסמכים חדשים לפני הפעלה (ברירת מחדל: 30)
  • תיקיית מסמכים -- היכן נמצאים מסמכי ה-SR-PTD (כולל דפוסי קבצים נתמכים, למשל SR-PTD*/task_doc*)
  • תיקיית Skills -- לאן להתקין (נתיב יעד פעיל; הרחבת ~ ונתיב ברירת מחדל לפי OS/ClaUi)
  • נתיב Python -- מיקום Python (ברירת מחדל: python)
  • נתיב הכלים -- מיקום ה-toolkit
  • תיקיית Workspace לצנרת -- תיקיה מבודדת ל-extractions/, clusters/, skills_out/ ולוגים
  • מצב אינטגרציה לצנרת -- Python API (מועדף) או run_pipeline.py עם Resume; create_skills.py ל-MVP/דיבוג בלבד
  • Resume אחרי כשל -- שימוש ב-.pipeline_progress.json כשזמין, או התחלה מחדש מה-staging בלי לאבד queue
  • Timeout / ביטול -- זמן ריצה מקסימלי, ומה קורה בלחיצת Cancel (סיום עדין / עצירה)
  • הרצה אוטומטית -- הרצה אוטומטית בהגעה לסף, או הודעה בלבד
  • AI לכפילויות -- שימוש ב-AI לזיהוי כפילויות לא ברורות

▶ שלבי יישום

שלב 1: תשתית
מונה + ledger למסמכים, הגדרות, מחוון בממשק, סריקה בסיסית, preflight (Python/toolkit/API key), ונעילה בין תהליכים/חלונות. מניח תשתית בטוחה לכל השאר.
שלב 2: אינטגרציית הצנרת
חיבור למערכת Python (עדיפות Python API / run_pipeline.py), Workspace מבודד, תמיכה ב-resume, הצגת התקדמות חיה, ופאנל מלא עם ניהול הרצות והיסטוריה.
שלב 3: כפילויות והתקנה
מנוע זיהוי כפילויות (traceability + תוכן + AI אופציונלי), השוואה מול Skills קיימים, התקנה מ-staging ליעד, גיבוי/rollback, וסיכום "מה השתנה".
שלב 4: ליטוש
הודעת סף, הרצה אוטומטית, כוונון ספי כפילויות עם נתוני אמת, תיעוד, בדיקות עומס/שגיאות/ביטול ושיפור UX.

⚠ סיכונים ומקרי קצה

Python לא מותקן
בדיקה בהפעלה. אם חסר -- הפיצ'ר מושבת עם הודעה ברורה.
מפתח API חסר
שלבים דטרמיניסטיים יכולים לרוץ, אבל לא מתבצעת סינתזה/התקנה מלאה. מוצגת הודעה ברורה, וה-queue לא מתאפס.
כשל באמצע הרצה
שמירת מצב ריצה + ארטיפקטים ב-workspace. אם משתמשים ב-run_pipeline.py -- Resume דרך .pipeline_progress.json; אחרת התחלה מחדש מה-staging. כשל נרשם בהיסטוריה.
ספירה כפולה של מסמכים
לא להסתמך על hash של שמות קבצים בלבד. להשתמש ב-fingerprint (path יחסי + mtime + size) או hash תוכן, ולעדכן סטטוס רק אחרי עיבוד/התקנה מוצלחים.
הרצה בו-זמנית ממספר חלונות
נעילה בין-תהליכית (lock file / named mutex) ולא רק isRunning בזיכרון. חלון שני רואה "הרצה בתהליך" וממתין/מציע Retry.
התקנה חלקית של Skills
התקנה מ-staging עם גיבוי לפני overwrite, כתיבה אטומית ככל האפשר, ו-rollback אם שלב התקנה נכשל באמצע.

❓ החלטות פתוחות

האם Skills מותקנים אוטומטית או דורשים אישור?
המלצה: ברירת מחדל "אישור נדרש" בהרצות הראשונות + אפשרות לעבור לאוטומטי. בכל מקרה: staging + גיבוי + rollback + סיכום "מה השתנה".
מתי לסרוק מסמכים חדשים?
המלצה: בהפעלת התוסף + בסיום כל סשן + רענון ידני. בלי file watcher בזמן אמת בגרסה הראשונה (מוריד מורכבות וכשלים).
איזו נקודת כניסה לצנרת לבחור?
המלצה: אינטגרציה דרך Python API (הכי נשלט) או run_pipeline.py עם Workspace ייעודי ו-Resume. להימנע מ-create_skills.py בפרודקשן בגלל timeout פר-פאזה וללא Resume.
מתי מאפסים את מונה/תור המסמכים?
המלצה: רק אחרי שריצה + בדיקות + התקנה הושלמו בהצלחה. כשל חלקי לא מוחק מסמכים מהתור.
