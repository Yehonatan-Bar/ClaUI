**תוכנית יישום מפורטת: מסלול מקביל ל־Codex (בלי לפגוע במסלול Claude שעובד)**

**החלטה ארכיטקטונית**
נלך על `Parallel Track` מלא ל־Codex, עם שכפול מכוון של חלקים בצד ההרחבה (extension), ולא על הכללה/ריפקטור של המסלול הקיים.

המשמעות:
- מסלול Claude נשאר עובד כמו היום כמעט ללא שינוי.
- מסלול Codex נכתב בנפרד, עם פרוטוקול/לוגיקה משלו.
- ה־UI המשותף נשאר ברובו זהה, עם התאמות קטנות בלבד.
- שכפול קוד הוא החלטה מודעת לטובת יציבות ומהירות יישום.

**עקרונות מחייבים**
1. לא משנים התנהגות קיימת של Claude מעבר לניתוב לפי provider.
2. לא מנסים "לאחד" פרוטוקולים בשלב ראשון.
3. מוסיפים `provider` כקונספט במעטפת בלבד (UI/state/store/commands).
4. מסלול Codex יעמוד לבד, גם אם יש קוד כפול.
5. פיצ'רים לא נתמכים ב־Codex יוסתרו/יושבתו במפורש, לא "ישברו בשקט".

---

**מטרה פונקציונלית (מה המשתמש יקבל)**
1. כפתור חדש `Codex` בסרגל התחתון.
2. בחירה של provider (`Claude` / `Codex`) כברירת מחדל לשיחות חדשות.
3. פתיחת טאב חדש של Codex עם UI דומה מאוד ל־Claude.
4. שליחת הודעות, קבלת תשובה, הרצת פקודות דרך Codex, ביטול, Resume בסיסי.
5. היסטוריה נפרדת/מאוחדת עם סימון provider.
6. מסלול Claude נשאר יציב ולא מושפע.

---

**לא־מטרות (MVP)**
1. לא "ממירים" session פעילה של Claude ל־Codex באותו thread.
2. לא עושים תאימות מלאה ל־Plan Approval Bar של Claude במסלול Codex.
3. לא עושים ריפקטור גנרי לכל שכבת ה־process/protocol.
4. לא מבטיחים parity מלא של streaming delta (ל־Codex אין כרגע אותו פורמט).

---

**הגדרת UX מדויקת לכפתור החדש**
**הכפתור**
- מיקום: ליד `ModelSelector` בסרגל התחתון (`StatusBar`).
- טקסט: `Codex`.
- מצב פעיל: מודגש אם ה־provider הנבחר הוא `codex`.
- tooltip:
  - אם provider נוכחי `claude`: `Switch default provider to Codex and open a new Codex tab`
  - אם provider נוכחי `codex`: `Codex is the current provider`

**התנהגות לחיצה**
1. אם אין session פעילה בטאב:
- הכפתור מגדיר provider = `codex`.
- לא פותח אוטומטית session אלא אם המשתמש לוחץ Start (לשמור על התנהגות עקבית).
- אופציה חלופית (אם רוצים UX מהיר): לפתוח טאב חדש ולהתחיל session לוגית. נבחר אחת בלבד לפני יישום. המלצה: לפתוח טאב חדש.

2. אם יש טאב Claude פעיל:
- הכפתור לא "מחליף" את הטאב הקיים.
- הוא עושה:
  - `provider` ברירת מחדל = `codex`
  - פותח טאב חדש במסלול Codex
- הטאב Claude נשאר פתוח וללא שינוי.

3. אם יש טאב Codex פעיל:
- הכפתור לא עושה כלום (או רק מאשר מצב).
- לא פותח טאב נוסף כדי למנוע accidental tab spam.

4. אם יש turn פעיל (busy):
- הכפתור disabled בטאב הנוכחי.
- לא מבצעים מעבר בזמן עיבוד.

הערה חשובה:
זה UX נכון מקצועית, כי מעבר provider הוא לא החלפת מודל. אין הבטחה להמשך אותו session cross-provider.

**מסלול Codex: התנהגות טכנית (מלא)**
**מודל סשן**
במסלול Codex הסשן הוא לוגי, לא process מתמשך.
- נשמור `threadId`
- כל turn ירוץ בתהליך חדש
- turn ראשון:
  - `codex exec --json ...`
- turn המשך:
  - `codex exec resume --json <threadId> ...`

**בניית פקודה (מחייבת)**
מומלץ להעביר prompt דרך stdin כדי להימנע מבעיות escaping/length:
- turn ראשון: `codex exec --json -C <cwd> [--model <model>] -`
- turn המשך: `codex exec resume --json <threadId> [--model <model>] -`

הערה:
לא להגדיר בשלב ראשון flags של sandbox/approval unless verified config keys. להשתמש ב־defaults של Codex כדי לא לשבור תאימות מקומית.

**Cancel**
- `cancelRequest` במסלול Codex = kill process tree של ה־turn הפעיל
- הסשן נשאר פתוח (`threadId` נשמר)
- אין auto-resume process כי אין process מתמשך

**Stop Session**
- מסיים סשן לוגי
- אם יש turn פעיל -> kill
- שולח `sessionEnded`
- משאיר היסטוריה ב־UI עד clear/new session (כמו היום)

**מיפוי אירועי Codex ל־UI הקיים (מפורט)**
מטרת המיפוי היא שה־React UI יעבוד כמעט בלי שינוי.

1. `thread.started`
- מקור session/thread id
- נשלח ל־webview:
  - `sessionStarted { sessionId: thread_id, model: configuredModelOrUnknown, provider: 'codex' }`
- נשמור ב־`SessionStore`

2. `turn.started`
- נשלח `processBusy: true`
- נייצר `messageStart` סינתטי רק כאשר מתחיל להיווצר assistant output (לא מיד)

3. `item.completed` מסוג `agent_message`
- אם אין `messageStart` פתוח -> לשלוח `messageStart` סינתטי
- לשלוח `assistantMessage` עם `ContentBlock[]` מסוג text בלבד בשלב ראשון
- `messageStop` סינתטי
- לא להסתמך על streaming deltas

4. `item.started/completed` מסוג `command_execution`
- נצבור telemetry פנימי עבור:
  - command string
  - exit code
  - aggregated output
- לצורך dashboard/vitals:
  - נתרגם ל־tool pseudo `Bash`
- אפשרות MVP:
  - לא להציג tool block בצ'אט
  - כן לשמור `bashCommands` ל־`turnComplete`
- אפשרות Phase 2:
  - לסנתז `toolUseStart/toolUseInput` ל־UI

5. `turn.completed`
- נשלח `costUpdate` עם:
  - `costUsd: 0`
  - `totalCostUsd: 0`
  - tokens מתוך `usage`
- נשלח `turnComplete`
- נשלח `processBusy: false`

6. `item.completed` מסוג `reasoning`
- MVP: לא מציגים ב־chat
- רק לוג פנימי

7. unknown items/events
- לוג בלבד, לא crash

**CodexConversationReader – תכנון מפורט**
**מקור נתונים**
- `~/.codex/sessions/YYYY/MM/DD/rollout-...-<threadId>.jsonl`

**איתור קובץ**
1. חיפוש לפי thread id בתוך שם הקובץ.
2. העדפה לקבצים לפי `mtime` (הכי חדש).
3. סינון לפי `cwd` אם קיים ב־`session_meta` (optional אבל מומלץ).

**Parsing (MVP)**
- לקרוא שורות JSONL
- לטפל ב:
  - `session_meta`
  - `response_item` כאשר `payload.type === 'message'`
- לכל `message`:
  - אם `role === 'user'` -> להפוך ל־`SerializedChatMessage` user
  - אם `role === 'assistant'` -> להפוך ל־assistant
  - להתעלם מ־`developer`/`system`
- המרה של content:
  - `input_text` / `output_text` -> `ContentBlock { type: 'text', text }`
- אם אין parse מלא:
  - fallback: להחזיר `[]` וללוג שגיאה (לא crash)
