23 בפברואר 2026

הבעיה

תהליך יצירת ה-Skills מורכב מ-8 שלבים. ארבעה מהם דורשים קריאות AI (העשרת תגיות, קיבוץ, מיזוג, וסינתזה). כיום שלבים אלו משתמשים בספריית Python שדורשת מפתח API נפרד של Anthropic (ANTHROPIC_API_KEY).

הבעיה: המשתמש צריך להגדיר מפתח API נפרד, מה שמסרבל את התהליך ויוצר תלות בחשבון נוסף. התהליך נכשל 3 פעמים בגלל בעיות קונפיגורציה שונות.

הפתרון

ClaUi כבר משתמש בקריאות CLI של Claude Code (לשיפור פרומפטים, מתן שמות, תרגום ועוד). קריאות אלו עוברות דרך החשבון הקיים של המשתמש - ללא צורך במפתח API נפרד.

הפתרון: נעביר את כל קריאות ה-AI של תהליך יצירת ה-Skills לעבור דרך אותו מנגנון CLI קיים. שלבים שלא דורשים AI ימשיכו לרוץ ב-Python כרגיל.

מבנה השלבים

8 שלבי התהליך, מחולקים לפי סוג הריצה:

  • Python B - חילוץ מידע מ-SR-PTD (עיבוד קבצים בלבד)
  • Python C.0-C.1 - יצירת כרטיסי מסמכים וקיבוץ ראשוני
  • CLI C.2 - העשרת תגיות באמצעות AI (Sonnet)
  • CLI C.3 - קיבוץ מתקדם באמצעות AI (Sonnet)
  • CLI C.4 - מיזוג בין-קבוצות באמצעות AI (Sonnet)
  • Python C.5 - בחירת נציגים (עיבוד קבצים בלבד)
  • Python Sanity - בדיקת תקינות
  • CLI D - סינתזת Skills באמצעות AI (Opus) - מקבילי!

תוכנית ביצוע

  1. יצירת מנגנון שיתופי לקריאות CLI (בסיס לכל שלבי ה-AI)
  2. יצירת מריץ Python לשלבים ללא AI (B, C.0-C.1, C.5, Sanity)
  3. מימוש שלב C.2 - העשרת תגיות (הפשוט ביותר, טוב לבדיקה ראשונה)
  4. מימוש שלבים C.3 ו-C.4 - קיבוץ ומיזוג
  5. מימוש שלב D - סינתזת Skills עם ריצה מקבילית
  6. חיבור כל השלבים למתזמר מרכזי חדש
  7. בדיקות מקצה לקצה - הרצת התהליך המלא
  8. עדכון תיעוד ו-SR-PTD

יתרונות

ביטול מפתח API - לא צריך יותר ANTHROPIC_API_KEY. התהליך עובד עם חשבון Claude Code הקיים.

האצת שלב D - סינתזת Skills רצה במקביל (עד 3 בו-זמנית) במקום אחד-אחד. זה השלב הכבד ביותר.

שליטה טובה יותר - כל שלב רץ בנפרד עם דיווח התקדמות מדויק, במקום תהליך Python אחד גדול.

תיקון באג - שלב C.4 היה מסומן כ"לא דורש API" למרות שדורש. בארכיטקטורה החדשה זה מטופל נכון.
