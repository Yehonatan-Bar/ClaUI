1. The Problem

The current StatusBar packs every feature into a single horizontal row. At full width (~1500px), this is what users see:

Active: 00:00:00 37 Trophy History Prompts Plans Feedback Consult Git * Claude Codex Happy Carry Model Opus 4.6 Permissions Full Access Aa Dashboard SkillDocs 4/50 ! Usage -> Vitals Settings In: 0 Out: 0
!
Core UX problems:
No hierarchy — Rarely-used "Font" sits next to critical "Model" selector. Everything competes for equal attention.
No grouping — Buttons are adjacent but have no visual boundary. "Claude/Codex/Happy/Carry" and "Model/Permissions" serve the same purpose (AI config) but look disconnected.
Cognitive overload — A new user must scan 25+ items to find anything. Power users must remember positions.
No progressive disclosure — Features you use once a week take the same space as features you use every minute.

2. Design Principles

1

Progressive Disclosure

Show only what users need right now. Everything else is one click away, organized logically.

2

Visual Hierarchy

Frequently used features are prominent and always visible. Rarely used ones are tucked into menus.

3

Semantic Grouping

Controls that answer the same question belong together. "What AI?" = one control. "Session tools?" = one menu.

4

Compound Controls

Merge related items into single multi-segment chips. Provider + Model + Permissions = one clickable element.

5

Information at a Glance

Passive metrics (time, usage, tokens) are always visible but compact. They inform without demanding attention.

6

Graceful Degradation

At narrow widths, groups collapse into fewer menus. At minimal width, one "Menu" button contains everything.

5b. Group Buttons (Session / Tools / View)

Three lightweight text buttons that open categorized dropdown panels. Similar to how VS Code's status bar items open quick-pick menus.

Keyboard shortcuts preserved: History (Ctrl+H), Plans (Ctrl+P), Git (Ctrl+G) still work directly — the dropdown is just an alternative discovery path. Power users never need to open the menus.

5c. Status Metrics (Right side)

Three passive indicators that stay visible without demanding attention:

T 12:34 — session time
72% — usage limit
In: 1.2k Out: 3.4k — tokens

The usage bar changes color: green (<60%), yellow (60-85%), red (>85%).

From 25+ visible controls to 7. The interface breathes. New users aren't overwhelmed.

1 click

Max Depth

Every feature is at most 1 click away. No features are hidden deeper than a single dropdown.

4 groups

Mental Model

Users learn 4 categories (AI, Session, Tools, View) instead of memorizing 25 button positions.

0

Features Removed

Every single feature is preserved. Nothing is lost — just better organized.

Migration strategy: Add tooltips during the first few sessions after the redesign. Each tooltip says "X moved here" pointing to the new group. This helps existing users find relocated features quickly. Consider a one-time "What's new" overlay on first launch.

8. Implementation Notes

Reuse Existing Infrastructure

The current codebase already has StatusBarGroupButton.tsx with dropdown logic, useStatusBarCollapse.ts with responsive breakpoints, and the dropdown styling infrastructure. The redesign builds on these patterns — it's reorganizing, not rewriting.

Key Changes

ComponentChangeEffort
StatusBar.tsx Restructure layout into AI Chip + 3 groups + metrics. Reuse StatusBarGroupButton. Medium
AIChip.tsx (new) Compound control combining provider buttons, model selector, and permissions selector. Medium
useStatusBarCollapse.ts Update breakpoints for new layout (fewer items = lower thresholds). Small
StatusBar.css Add AI Chip styles, update group button styles, add metric styles. Small
All existing features No change to logic. Only their "container" moves from inline to dropdown. None

What NOT to Change

  • The InputArea — it works well as-is
  • Feature logic and state — only the visual container changes
  • Keyboard shortcuts — they bypass the UI entirely
  • The overlay panels (Dashboard, History, Plans) — they open the same way
