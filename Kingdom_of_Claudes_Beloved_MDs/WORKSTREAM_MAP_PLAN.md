# Workstream Map - Development Plan

## Executive Summary

A visual "subway map" dashboard that transforms scattered Claude sessions into coherent workstream narratives per project. Sessions are grouped by AI classification into logical work flows (feature, bug, research, refactor, etc.) and displayed as colored lines with stations representing significant events.

// === Workstream Identity ===

export type WorkstreamType =

  | 'bug-fix' 
  | 'research' 
  | 'refactor' 
  | 'infrastructure' 
  | 'experiment-abandoned' 
  | 'uncategorized';

export type WorkstreamStatus = 
  | 'active'       // blue
  | 'completed'    // green  
  | 'blocked'      // red
  | 'uncertain'    // yellow
  | 'research'     // purple
  | 'abandoned'    // gray
  | 'planning';    // light blue

export type StationType = 
  | 'session'           // circle
  | 'decision'          // diamond
  | 'code-change'       // square
  | 'problem'           // triangle
  | 'milestone'         // star
  | 'failure'           // x
  | 'uncertainty'       // question mark
  | 'blocker'           // lock
  | 'direction-change'  // curved arrow
  | 'merge-point'       // junction
  | 'plan-step';        // outlined circle

export type LineTexture = 
  | 'solid'      // confident classification
  | 'dashed'     // uncertain classification
  | 'blurred'    // unstructured work
  | 'broken';    // blocked/failed

// === Workstream Model ===

export interface Workstream {
  id: string;                      // uuid
  projectId: string;               // workspace path hash
  label: string;                   // e.g., "Auth middleware rewrite"
  goal?: string;                   // one-line purpose
  type: WorkstreamType;
  status: WorkstreamStatus;
  color: string;                   // hex, from palette
  
  // Relationships
  sessionIds: string[];            // ordered by time
  relatedWorkstreamIds: string[];  // linked streams
  parentWorkstreamId?: string;     // if split from another
  mergedIntoWorkstreamId?: string; // if merged
  
  // AI Metadata
  confidence: number;              // 0-1, how sure the classification is
  autoGenerated: boolean;          // true if AI-created, false if user-created
  
  // Temporal
  startedAt: string;               // ISO date
  lastActivityAt: string;          // ISO date
  completedAt?: string;            // ISO date
  
  // Plan association
  planId?: string;                 // reference to a plan doc if exists
  planSteps?: PlanStep[];          // plan overlay data
  
  // Metrics
  totalSessions: number;
  totalCostUsd: number;
  totalTurns: number;
  filesModified: string[];         // unique file paths touched
  
  // Ordering
  order: number;                   // visual sort order
}

export interface Station {
  id: string;                      // uuid
  workstreamId: string;
  type: StationType;
  label: string;                   // short description (max 60 chars)
  description?: string;            // optional longer detail
  
  // Source
  sessionId?: string;              // originating session
  turnIndex?: number;              // specific turn within session
  
  // Position on the line (logical order, not pixel coords)
  order: number;
  
  // State
  status: 'completed' | 'partial' | 'failed' | 'pending' | 'skipped';
  
  // Visual
  color?: string;                  // override for specific stations
  size?: 'small' | 'medium' | 'large'; // based on significance
  
  // Temporal
  timestamp: string;               // ISO date
  
  // Relations
  splitToWorkstreamIds?: string[]; // if this station is a split point
  mergedFromWorkstreamIds?: string[]; // if this is a merge point
}

export interface PlanStep {
  id: string;
  label: string;
  order: number;
  status: 'completed' | 'partial' | 'failed' | 'pending' | 'skipped';
  stationId?: string;              // linked station if executed
  deviationNote?: string;          // why reality diverged
}

export interface Split {
  id: string;
  fromWorkstreamId: string;
  toWorkstreamIds: string[];
  stationId: string;               // the station where split occurs
  reason: string;                  // short explanation
  timestamp: string;
}

export interface Merge {
  id: string;
  fromWorkstreamIds: string[];
  toWorkstreamId: string;
  stationId: string;
  reason: string;
  timestamp: string;
}

// === Project-Level Model ===

export interface ProjectMapState {
  projectId: string;
  projectLabel: string;
  workstreams: Workstream[];
  stations: Station[];
  splits: Split[];
  merges: Merge[];
  lastClassifiedAt: string;        // when AI last ran
  userEdits: UserEdit[];           // track manual changes for AI not to override
}

export interface UserEdit {
  id: string;
  type: 'move-session' | 'merge-workstreams' | 'split-workstream'

  timestamp: string;
  details: Record<string, unknown>;
}

// === Multi-Project View ===

export interface ProjectDot {
  projectId: string;
  label: string;
  workspacePath: string;
  activeWorkstreams: number;

  overallStatus: WorkstreamStatus;
  lastActivity: string;            // ISO date
  recentActivityLevel: number;     // 0-1, for dot size/glow
  needsAttention: boolean;         // for pulsing dot
}

### Phase 1: AI Classification Engine
**Goal:** Group sessions into workstreams automatically.
**Effort:** 5-7 days

#### 1.1 WorkstreamClassifier service

**File:** NEW `src/extension/workstream/WorkstreamClassifier.ts`

Responsibilities:
- Take all `SessionSummary[]` for a project
- Group them into workstreams based on signals:
  - File overlap (sessions touching same files = likely same workstream)
  - Topic similarity (firstPrompt, summary text)
  - Temporal proximity (sessions close in time on same topic)
  - Git branch (same branch = same workstream)
  - Task type continuity (bug-fix sessions clustered together)
  - Explicit references ("continuing from...", "back to the auth work...")

**Classification approach (2-pass):**

**Pass 1 - Heuristic pre-clustering:**
```typescript
// Group by git branch (strongest signal)
// Group by file overlap > 50% 
// Group by time proximity + same task type
// Everything else = uncategorized
```

**Pass 2 - AI refinement (Sonnet 4.6):**
- Send pre-clusters + session summaries to Claude Sonnet 4.6 (default model for all classification)
- Prompt: "Given these session summaries, confirm/adjust the workstream groupings. For each workstream, provide: label, type, goal, confidence."
- Respect `userEdits` - never reassign sessions the user manually placed

**AI Prompt template:**
```
You are analyzing coding sessions for a software project.
Group these sessions into logical workstreams (ongoing threads of work).

Sessions:
[{id, firstPrompt, summary, filesModified, gitBranch, taskType, date}...]

Current groupings (from heuristics):
[{workstreamLabel, sessionIds}...]

User manual assignments (DO NOT change these):
[{sessionId, workstreamId}...]

Output JSON:
{
  workstreams: [{
    label: string,       // short descriptive name
    type: WorkstreamType,
    goal: string,        // one sentence
    sessionIds: string[],
    confidence: number   // 0-1
  }]
}
```

#### 1.2 StationExtractor service

**File:** NEW `src/extension/workstream/StationExtractor.ts`

For each session within a workstream, extract significant "stations":
- Session start (always a station of type 'session')
- Major decisions (detected from conversation - "let's go with approach X")
- Failures (turns with `taskOutcome: 'failed'`)
- Blockers (turns with `bugRepeat: 'third-plus'` or explicit blocker language)
- Milestones (tests passing, PRs merged, deploys)
- Direction changes (when taskType shifts mid-session)

**Approach:** For each session, use Sonnet 4.6 to analyze the conversation summary + key turns and extract 2-5 stations. Cap at max 5 stations per session to keep the map readable.

#### 1.3 WorkstreamStore (persistence)

**File:** NEW `src/extension/workstream/WorkstreamStore.ts`

- Backed by `workspaceState` (project-scoped, like ProjectAnalyticsStore)
- Stores: `ProjectMapState` (workstreams, stations, splits, merges, userEdits)
- Write queue pattern (same as ProjectAnalyticsStore)
- Max storage: 50 workstreams per project, 500 stations total
- Garbage collection: archive completed workstreams older than 90 days

#### 1.4 Classification trigger & incremental updates

**File:** NEW `src/extension/workstream/WorkstreamManager.ts`

The orchestrator that ties it together:
- **On session end:** Run incremental classification (just the new session)
- **On first activation:** Full classification of all stored sessions
- **On user request:** Re-classify everything (respecting user edits)
- **Debounce:** Don't re-classify more than once per 5 minutes
- Emits events for the webview to consume

#### 2.4 Layout Algorithm

**File:** NEW `src/webview/components/WorkstreamMap/layout.ts`

Subway map layout algorithm:
1. Sort workstreams by `startedAt`
2. Assign vertical "lanes" (avoid crossing lines)
3. Place stations along each line at proportional horizontal positions
4. Route split/merge junctions with smooth Bezier curves
5. Handle line thickness (based on session count)
6. Handle line texture (based on confidence)

Key constraints:
- Lines should be mostly horizontal (left-to-right = time)
- Vertical space between lines = readability
- Splits angle downward, merges angle upward
- Stations are evenly spaced (semantic, not temporal)

#### 2.5 Visual Encoding

Colors (from existing VS Code theme vars + custom):
```css
--workstream-active: var(--vscode-charts-blue);
--workstream-completed: var(--vscode-charts-green);
--workstream-blocked: var(--vscode-charts-red);
--workstream-uncertain: var(--vscode-charts-yellow);
--workstream-research: var(--vscode-charts-purple);
--workstream-abandoned: var(--vscode-descriptionForeground);
```

Station shapes (SVG):
- Circle = session
- Diamond (rotated square) = decision
- Square = code change
- Triangle = problem/risk
- Star (5-point) = milestone
- X = failure
- ? (circle with ?) = uncertainty
- Lock icon = blocker

Line properties:
- Stroke-width: 2px (thin/few sessions) to 6px (thick/many sessions)
- Stroke-dasharray: none (solid/confident), "8,4" (dashed/uncertain), custom filter (blurred)
- Opacity: 1.0 (active), 0.6 (completed), 0.3 (abandoned)

#### 2.6 Zoom Implementation

Uses CSS transform + SVG viewBox manipulation:
- **Projects view:** Simple vertical dot layout (no SVG complexity)
- **Project view:** Full subway map, animated zoom transition from dot
- **Workstream view:** Single line expanded, other lines faded/hidden
- **Detail view:** Station details visible, side panel with sessions list

Zoom transitions: CSS `transform: scale()` + `translate()` with 300ms ease-out.

### Phase 3: Interaction Layer
**Goal:** Make the map interactive and navigable.
**Effort:** 5-6 days

#### 3.1 Click/Hover Interactions

- **Hover station:** Show tooltip (label, timestamp, status)
- **Click station:** Zoom to detail, show associated session/turn info
- **Hover line:** Highlight full workstream, show workstream name

- **Right-click station:** Context menu (move, delete, mark as milestone)

#### 3.2 Focus Mode (single workstream)

When a workstream is focused:
- Other lines fade to 10% opacity
- The focused line expands vertically
- Stations grow larger, labels become visible
- Side panel shows:
  - Workstream metadata (name, type, status, goal)
  - Session list with summaries
  - Files modified
  - Key decisions
  - Blockers/problems
  - Progress vs plan (if plan exists)
  - "Next recommended action"

#### 3.3 Navigation

- Mouse wheel / pinch = zoom in/out
- Click+drag on background = pan
- Breadcrumb trail: Projects > ProjectName > WorkstreamName
- Back button / Escape = zoom out one level
- Keyboard: Arrow keys to navigate stations, Enter to zoom in, Escape to zoom out

#### 4.3 Dual-Line Visualization

**File:** `PlanOverlayLine.tsx`

When plan exists:
- Thin, faded line = planned path (straight, evenly spaced plan steps)
- Thick, solid line = actual path (with deviations)
- Where lines converge = "on track"
- Where lines diverge = "deviated"
- Color coding on plan steps:
  - Green circle = completed step
  - Yellow circle = partially done
  - Red circle = failed
  - Gray circle = not started
  - Dashed connection where actual path left planned path

When user enters Resolve mode:
- Lines become draggable

## Key Technical Decisions

### 1. SVG vs Canvas
**Decision: SVG**
- Accessibility (DOM nodes, aria attributes)
- CSS transitions for animations
- Easy hit testing (native DOM events)
- Scalable without pixelation
- Trade-off: Performance may degrade at >100 stations; mitigated by zoom (only render visible elements)

### 2. Layout Strategy
**Decision: Force-directed with constraints**
- Time flows left-to-right (x-axis)
- Workstreams assigned to lanes (y-axis) 
- Minimize crossings using a modified Sugiyama algorithm (simplified)
- Stations evenly spaced along each line (semantic distance, not temporal)
- Fall back to simple vertical stacking if fewer than 5 workstreams

### 3. Classification Trigger
**Decision: Incremental on session end + full on demand**
- Don't classify on every turn (too expensive)
- Classify new sessions at session end against existing workstreams
- Full re-classification only on user request or first activation
- Respect user edits always (never override manual placements)

### 4. Storage Limits
**Decision: Workspace-scoped with caps**
- Max 50 workstreams per project
- Max 500 stations per project
- Archive completed workstreams after 90 days to `archivedWorkstreams` key
- Total storage estimate: ~100KB per project (well within VS Code Memento limits)

### 5. Multi-Project Support
**Decision: Deferred to Phase 6+**
- Initial implementation: single-project view only (current workspace)
- Multi-project "dots" view requires reading other workspace states
- Can be added later using SessionDiscovery's cross-workspace scan

### 6. AI Model for Classification
**Decision: Sonnet 4.6 as default**
- Sonnet 4.6 for all classification tasks (best accuracy-to-cost ratio for structured analysis)
- Incremental classification: 1 call per session end (classify new session into existing workstreams)
- Full re-classification: batch all sessions in one call (efficient prompt caching)
- Station extraction: 1 call per session (extract 2-5 key events)
- Cap AI calls: max 3 per session end, max 10 per full reclassification
- Total added cost per session: ~$0.005-0.015

## Risk Factors

| Risk | Mitigation |
|------|-----------|
| AI classification accuracy | Confidence scores + easy manual override |
| Performance with many sessions | Lazy loading, only classify on demand, pagination |
| SVG performance with complex maps | Virtual rendering, only draw visible elements |
| Storage limits (VS Code Memento) | Aggressive archival, cap workstreams/stations |
| Cost of AI calls | Haiku (cheap), strict caps, incremental-only by default |
| Layout algorithm complexity | Start simple (vertical lanes), iterate |
| User confusion with auto-classification | Always show confidence, easy Resolve mode |
